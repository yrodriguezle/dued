## Glossary

- **SUT**: System Under Test
- **MUST/SHOULD/MAY**: RFC 2119 keywords indicating requirement strength
- **Coverage target**: Minimum line coverage percentage measured by coverlet (backend) or istanbul/v8 (frontend)

### 1.2 JwtHelper Tests

**SUT**: `backend/Services/Jwt/JwtHelper.cs`
**Support**: `backend/Services/Jwt/SecurityKeyType.cs`

#### REQ-1.2.1: Token Generation

- Tests MUST verify that `GenerateToken` produces a valid JWT string (three Base64URL segments separated by dots)
- Tests MUST verify that the generated token contains the expected claims (user ID, username, role)
- Tests MUST verify that the token expiry is set correctly based on configuration
- Tests SHOULD verify that different users produce different tokens

**Scenario: Generate access token with valid claims**
```
Given a user with ID=1, username="admin", role="Admin"
When I call GenerateToken with these user details
Then the returned string MUST be a valid JWT
And decoding the token MUST yield claim "sub" = "1"
And decoding the token MUST yield claim "unique_name" = "admin"
And decoding the token MUST yield claim "role" = "Admin"
And the token "exp" claim MUST be within ±5 seconds of (now + configured expiry)
```

#### REQ-1.2.2: Token Validation

- Tests MUST verify that a freshly generated token passes validation
- Tests MUST verify that a token with a tampered signature fails validation
- Tests MUST verify that a token signed with a different key fails validation
- Tests MUST verify that an expired token fails validation

**Scenario: Reject tampered token**
```
Given a valid JWT generated by JwtHelper
When I modify one character in the signature segment
And I call ValidateToken with the tampered token
Then validation MUST return false or throw a SecurityTokenException
```

**Scenario: Reject expired token**
```
Given a JWT generated with an expiry of 1 second
When I wait 2 seconds
And I call ValidateToken
Then validation MUST fail
```

#### REQ-1.2.3: Claims Extraction

- Tests MUST verify that claims can be extracted from a valid token
- Tests MUST verify that extracting claims from an invalid token returns null or throws

**Scenario: Extract claims from valid token**
```
Given a valid JWT containing claim "sub" = "42"
When I extract the ClaimsPrincipal from the token
Then the principal MUST contain a claim of type "sub" with value "42"
```

### 1.3 PasswordService Tests

**SUT**: `backend/Services/HashPassword/PasswordService.cs`

#### REQ-1.3.1: Password Hashing

- Tests MUST verify that hashing a password produces a non-null, non-empty string
- Tests MUST verify that hashing the same password twice produces different hashes (salt randomization)
- Tests MUST verify that the hash format is recognizable (BCrypt or PBKDF2 prefix)

**Scenario: Same password produces different hashes**
```
Given the password "SecureP@ss123"
When I hash it twice using PasswordService
Then hash1 MUST NOT equal hash2
And both hashes MUST be non-empty strings
```

#### REQ-1.3.2: Password Verification

- Tests MUST verify that a password matches its own hash
- Tests MUST verify that an incorrect password does NOT match a hash
- Tests MUST verify behavior with empty string input
- Tests SHOULD verify behavior with null input (expect ArgumentNullException or false)

**Scenario: Correct password verifies**
```
Given the password "MyPassword1!" hashed to storedHash
When I verify "MyPassword1!" against storedHash
Then the result MUST be true
```

**Scenario: Wrong password rejected**
```
Given the password "MyPassword1!" hashed to storedHash
When I verify "WrongPassword" against storedHash
Then the result MUST be false
```

#### REQ-1.3.3: Edge Cases

- Tests SHOULD verify behavior with very long passwords (>1000 characters)
- Tests SHOULD verify behavior with unicode characters in passwords
- Tests MAY verify performance stays under 500ms for a single hash operation

### 1.4 AuthController Tests

**SUT**: `backend/Controllers/AuthController.cs`

#### REQ-1.4.1: Sign-In Flow

- Tests MUST verify that valid credentials return 200 with a JWT and set a refresh token cookie
- Tests MUST verify that invalid credentials return 401
- Tests MUST verify that a missing username or password returns 400
- Tests SHOULD verify that the response includes the expected token structure

**Scenario: Successful sign-in**
```
Given a registered user with username="cashier" and password="Test123!"
When I POST /api/auth/signin with these credentials
Then the response status MUST be 200
And the response body MUST contain a non-empty "accessToken" field
And a "refreshToken" httpOnly cookie MUST be set in the response
```

**Scenario: Invalid credentials**
```
Given a registered user with username="cashier" and password="Test123!"
When I POST /api/auth/signin with username="cashier" and password="WrongPass"
Then the response status MUST be 401
And no refreshToken cookie MUST be set
```

#### REQ-1.4.2: Token Refresh

- Tests MUST verify that a valid refresh token cookie returns a new access token
- Tests MUST verify that an expired or invalid refresh token returns 401
- Tests MUST verify refresh token rotation (old token invalidated after use)

**Scenario: Refresh with valid token**
```
Given a valid refreshToken cookie from a previous sign-in
When I POST /api/auth/refresh with that cookie
Then the response status MUST be 200
And a new accessToken MUST be returned
And a new refreshToken cookie MUST replace the old one
```

#### REQ-1.4.3: Sign-Out

- Tests MUST verify that sign-out clears the refresh token cookie
- Tests MUST verify that the invalidated refresh token cannot be reused

**Scenario: Sign-out clears session**
```
Given an authenticated session with a valid refreshToken cookie
When I POST /api/auth/signout
Then the response status MUST be 200
And the refreshToken cookie MUST be cleared (expired or deleted)
When I POST /api/auth/refresh with the old refreshToken
Then the response status MUST be 401
```

### 1.5 AuthRateLimitMiddleware Tests

**SUT**: `backend/Middleware/AuthRateLimitMiddleware.cs`

#### REQ-1.5.1: Rate Limiting Enforcement

- Tests MUST verify that requests within the rate limit are allowed (200 or pass-through)
- Tests MUST verify that requests exceeding the rate limit are rejected (429)
- Tests MUST verify that the rate limit window resets after the configured time period
- Tests SHOULD verify that rate limiting is per-IP or per-client as designed

**Scenario: Requests within limit pass through**
```
Given a rate limit of 10 requests per 60 seconds
When I send 10 requests from the same IP within 60 seconds
Then all 10 requests MUST receive pass-through (not 429)
```

**Scenario: Exceed rate limit**
```
Given a rate limit of 10 requests per 60 seconds
When I send 11 requests from the same IP within 60 seconds
Then the 11th request MUST receive status 429 (Too Many Requests)
```

**Scenario: Rate limit window resets**
```
Given the rate limit window has expired
When I send a new request from a previously rate-limited IP
Then the request MUST be allowed through
```

## Cross-Cutting Requirements

### REQ-CC.1: Test Isolation

- Every test MUST be independent — no shared mutable state between tests
- Backend tests MUST create a fresh InMemory DbContext per test (or per test class with cleanup)
- Frontend tests MUST not depend on global DOM state from previous tests

### REQ-CC.2: No Production Code Regression

- Adding tests MUST NOT change any production code behavior
- Minor DI adjustments (extracting interfaces) are acceptable but MUST be backward-compatible
- All existing 103 frontend tests MUST continue to pass after changes

### REQ-CC.3: CI Integration

- `dotnet test` from `backend/` MUST run all backend tests
- `npm test` (or `npx vitest run`) from `duedgusto/` MUST run all frontend tests
- Both commands MUST exit with code 0 when all tests pass
- Coverage reports SHOULD be generated in CI-consumable format (lcov, cobertura)

### REQ-CC.4: Coverage Thresholds

| Scope | Metric | Target |
|-------|--------|--------|
| Phase 1 target files | Line coverage | >= 90% |
| Phase 2 target files | Line coverage | >= 70% |
| Phase 3 stores | Line coverage | >= 70% |
| Phase 3 UI components | Line coverage | >= 50% |
| Phase 4 target files | Line coverage | >= 50% |
| Backend overall | Line coverage | >= 60% |
| Frontend overall | Line coverage | >= 55% |

### REQ-CC.5: Test Naming Convention

- Backend: `[MethodName]_[Scenario]_[ExpectedResult]` (e.g., `GenerateToken_ValidUser_ReturnsJwtWithCorrectClaims`)
- Frontend: descriptive `it("should ...")` or `test("...")` style matching existing patterns

### REQ-CC.6: Mocking Strategy

- Backend: Use Moq for service dependencies; EF Core InMemory for database; avoid mocking EF directly
- Frontend: Use MockedProvider for Apollo queries/mutations; vi.mock for module mocking; @testing-library/user-event for interactions
- MUST NOT mock the SUT itself — only its dependencies
